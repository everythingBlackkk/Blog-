Understanding Asymmetric Encryption with Simple Examples

Grab your coffee and let's dive into Asymmetric Encryption with a few easy-to-understand examples.

How Does Asymmetric Encryption Work?

Imagine you're managing a server, and instead of logging in with a password that could be stolen, you use a key. By adding a few security measures, you're much more secure than before when you were using just a password.

Let's start with an example of connecting to a server using SSH and AES with keys.

Step 1: Generating Key Pairs

If you're connecting to the server for the first time, you need to generate two keys (a public key and a private key). This can be done with the following command: 

ssh-keygen -t rsa

You can add a passphrase for extra security, or you can skip this step.

  • Private key: This will be saved in /home/user/.ssh/id_rsa.
  • Public key: This will be saved in /home/black/.ssh/id_rsa.pub.

Step 2: Sending the Public Key to the Server

To send your public key to the server, use the following command: 

ssh-copy-id blackserver@0.0.0.0

This command sends the public key to the server, which will store it in a file called ~/.ssh/authorized_keys. The server now recognizes that anyone with the corresponding private key can connect.

Step 3: Authentication Process

When you try to connect to the server via SSH, the server sends you an encrypted message using the public key you initially sent. Your device will use your private key to decrypt the message and then send it back to the server. The server then checks the decrypted message with the private key. If the message is correct and properly decrypted, the server confirms that you are indeed the owner of the private key and allows you to connect.

Step 4: Session Key

After the verification process, something called a Session Key is used.

Why is a "Session Key" used?

The idea here is that there needs to be a fast data exchange channel between the server and the client, and to achieve this, they agree on a shared session key.

What is a Session Key?

It is a Symmetric Key generated randomly and is used for encrypting and decrypting the data that will be sent during that session only.

How is the Session Key exchanged?

The key is usually generated by the client and then encrypted with the server's public key. After that, the server uses its private key to decrypt the session key. From this point on, the client and server use the session key for encryption and decryption.

Another Example from Your Friend "everythingBlackkk" in Python:


from Cryptodome.PublicKey import RSA
from Cryptodome.Cipher import PKCS1_OAEP

# 1. Generate a pair of keys (public and private)
key_pair = RSA.generate(2048)
public_key = key_pair.publickey()

# 2. Print the keys (just for illustration)
print("Public Key:")
print(public_key.export_key().decode())
print("\nPrivate Key:")
print(key_pair.export_key().decode())

# 3. Message to be encrypted
message = "Hello , i'm everythingBlackkk."

# 4. Encrypt the message using the public key
cipher = PKCS1_OAEP.new(public_key)
encrypted_message = cipher.encrypt(message.encode())

print("\nEncrypted Message:")
print(encrypted_message)

# 5. Decrypt the message using the private key
decrypt_cipher = PKCS1_OAEP.new(key_pair)
decrypted_message = decrypt_cipher.decrypt(encrypted_message)

print("\nDecrypted Message:")
print(decrypted_message.decode())

Some Questions

1. Isn't there a risk if my public key gets leaked?

Actually, no. Imagine you have a safe where you keep your money. Everyone who knows you knows where the safe is, but only you have the key to open it and take the money.

2. Why use Symmetric?

Symmetric encryption is much faster than Asymmetric encryption, but we use Asymmetric encryption initially for identity verification.
